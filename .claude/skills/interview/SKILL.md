# 요구사항 인터뷰

## 목표
사용자와의 대화를 통해 요구사항을 체계적으로 수집하고 정리한다.

## 트리거
- "새 기능을 만들고 싶어요"
- "요구사항 정리해줘"
- "프로젝트 시작할래요"
- 사용자가 기능 구현을 요청할 때

## 프로세스

### 1. 경청
사용자의 초기 아이디어를 충분히 경청한다. 이 단계에서는 판단하지 않고 모든 내용을 기록한다.

### 2. 탐색 질문
다음 질문들을 순차적으로 진행:

```
Q1: 비즈니스 목표
"이 기능의 비즈니스 목표가 무엇인가요?"
"어떤 문제를 해결하고 싶으신가요?"

Q2: 사용자
"주요 사용자는 누구인가요?"
"사용자의 페르소나를 설명해주세요."

Q3: 핵심 기능
"핵심 기능은 무엇인가요?"
"사용자가 할 수 있는 주요 행동은?"

Q4: 기술적 제약
"기술적 제약사항이 있나요?"
"특정 기술 스택을 사용해야 하나요?"
"일정 제약이 있나요?"
```

### 3. MoSCoW 분류
수집된 요구사항을 우선순위별로 분류:

| 분류 | 설명 | 예시 |
|------|------|------|
| **Must have** | 필수 기능, 없으면 서비스 불가 | 회원가입, 로그인 |
| **Should have** | 중요 기능, 있으면 좋음 | 이메일 인증 |
| **Could have** | 선택 기능, 나중에 추가 가능 | 소셜 로그인 |
| **Won't have** | 이번 버전 제외 | SSO 연동 |

### 4. 요구사항 초안 작성
질문 결과를 바탕으로 요구사항 초안 작성

## 출력 파일

### requirements-draft.md
```markdown
# 요구사항 초안

## 프로젝트명
[프로젝트 이름]

## 비즈니스 목표
- [목표 1]
- [목표 2]

## 사용자 페르소나
- [페르소나 1]: [설명]
- [페르소나 2]: [설명]

## 기능 요구사항

### Must have
- [ ] [요구사항 1]
- [ ] [요구사항 2]

### Should have
- [ ] [요구사항 3]

### Could have
- [ ] [요구사항 4]

### Won't have (이번 버전)
- [ ] [요구사항 5]

## 비기능 요구사항
- 성능: [요구사항]
- 보안: [요구사항]
- 확장성: [요구사항]

## 기술적 제약
- [제약 1]
- [제약 2]
```
